## 핵심 문제의식

> AI가 코드를 생성하는 시대,
> GitHub 커밋 수나 프로필만으로는 개발자의 **진짜 전문성과 기여 영향력**을 판단할 수 없다.

- AI 도구(Copilot, Cursor 등) 사용이 권장되는 시대 → 코드 자체보다 **판단력과 영향력**이 진짜 역량
- 커밋 수, 스타 수 같은 단순 지표는 실제 기여의 질을 반영하지 못함
- 기존 GitHub 분석 도구들은 "얼마나 많이 했는지"만 봄 → **실질적 기여 영향력은 측정 못 함**
- 분석 결과가 웹페이지로 끝나 위조가 가능하고 제3자가 신뢰하기 어려움

---

## 핵심 포지셔닝

> **"GitHub 분석기"가 아닌 "AI Professional Intelligence Engine"**
>
> AI를 얼마나 잘 활용해서 **실질적인 기여 영향력을 만들어냈는가**를 측정하고 증명한다.

| 기존 GitHub 분석 서비스 | OSSVerify |
|------------------------|-----------|
| 커밋 수 · 스타 수 집계 | 실질적 기여 영향력 측정 |
| 점수만 제공 | 점수 + 근거 제시 (Explainable AI) |
| 웹페이지로 끝남 (위조 가능) | VC 발급 → 제3자 검증 가능 |
| SDK 미제공 | SDK로 외부 서비스 바로 연동 |

## 핵심 흐름

```
GitHub 기여 데이터 수집
(Commit / PR / Issue / Code Review / 기술 스택)
        ↓
실질적 기여 영향력 분석
- PR이 실제로 머지됐는가
- 코드리뷰에서 의미 있는 피드백을 줬는가
- Maintainer가 승인했는가
- 프로젝트 방향 결정에 기여했는가
        ↓
전문 분야 추론
(단순 언어 사용량이 아닌 기여 맥락으로 판단)
        ↓
Explainable AI — 평가 근거 생성
        ↓
Professional Profile 생성
        ↓
W3C DID 기반 Verifiable Credential 발급
        ↓
블록체인 저장 (위변조 방지)
        ↓
SDK → 채용 / 프리랜서 / 인재관리 플랫폼
```

## 기능 목록

### 1. GitHub 활동 수집
- Commit
- Pull Request
- Issue
- Code Review
- Repository 정보
- 사용 언어 및 기술 스택

### 2. 실질적 기여 영향력 분석 ← 핵심 기능
단순 수치가 아닌 기여의 **질과 영향력**을 측정

두 가지 케이스를 모두 분석한다.

**케이스 1 — 타인 오픈소스 프로젝트 기여자로서의 역량**

| 측정 항목 | 설명 |
|----------|------|
| PR 머지율 | 제안한 PR이 실제로 반영됐는가 |
| 코드리뷰 품질 | 리뷰 코멘트가 실제 코드 개선으로 이어졌는가 |
| Maintainer 승인 | 프로젝트 핵심 관리자가 기여를 인정했는가 |
| 프로젝트 규모 | 기여한 프로젝트의 영향력 (스타 수, 포크 수) |
| 기여 지속성 | 단발성인지 장기적으로 기여했는지 |
| 이슈 해결률 | 제기한 이슈가 실제 해결로 이어졌는가 |

**케이스 2 — 본인 오픈소스 프로젝트 Maintainer로서의 역량**

| 측정 항목 | 설명 |
|----------|------|
| 프로젝트 채택률 | 스타 수, 포크 수, 다운로드 수 |
| 커뮤니티 활성도 | 외부 기여자가 얼마나 참여하는가 |
| 이슈 응답 속도 | 제기된 이슈에 얼마나 빠르게 대응하는가 |
| PR 리뷰 품질 | 외부 기여자 PR을 얼마나 잘 리뷰하는가 |
| 릴리즈 지속성 | 꾸준히 유지보수되고 있는가 |
| 문서화 수준 | README, 기여 가이드 등이 잘 갖춰져 있는가 |

**최종 프로파일은 두 케이스를 종합하여 산출**
- 기여자 역량 + Maintainer 역량을 합산한 종합 전문성 프로파일 생성

### 3. AI 전문성 분석
단순 언어 사용량이 아닌 기여 맥락으로 전문 분야 추론

| 분야 |
|------|
| Backend |
| Frontend |
| AI/ML |
| DevOps |
| Cloud |
| Security |
| Blockchain |

### 4. Explainable AI ← 차별점
점수만 주는 게 아니라 **평가 근거를 함께 제시**

```
Backend 91점
이유:
- Spring Boot 프로젝트 5개에 장기 기여
- REST API 관련 PR 34건 머지 완료
- Backend Code Review 51회 (코드 개선 반영률 78%)
- 2년 이상 지속 기여
```

### 5. Professional Profile 자동 생성

```
김동근

전문 분야     Backend / Security
주요 기술     Spring · Java · Docker · MySQL
기여 영향력   ★★★★☆
활동성        높음
```

### 6. DID 기반 VC 발급
- W3C DID 표준 기반 Verifiable Credential 발급
- 분석 결과를 검증 가능한 형태로 변환

### 7. VC 검증 API
- `verifyCredential()` 로 진위 · 위변조 여부 확인
- 블록체인에 저장하여 무결성 보장

### 8. SDK 제공
채용 플랫폼, 프리랜서 플랫폼, 사내 인재 관리 시스템 등에 바로 연동 가능

```
analyzeDeveloper()       // GitHub 분석 + 영향력 측정
getProfessionalProfile() // 프로필 조회
issueCredential()        // VC 발급
verifyCredential()       // VC 검증
```

## AI 모델 구성 (확정)

앙상블 방식 — 각 역할에 최적화된 모델을 조합하여 최종 점수 산출

| 역할 | 모델 | 학습 여부 | 우선순위 |
|------|------|----------|----------|
| 전문 분야 추론 | BERT 파인튜닝 | 직접 학습 | 1순위 |
| 영향력 계산 | PageRank | 직접 학습 | 2순위 |
| Explainable AI 근거 생성 | LLM (API 활용) | 학습 불필요 | 3순위 |
| 관계 구조 분석 | GNN | 직접 학습 | 4순위 |
| 최종 점수 산출 | XGBoost 앙상블 | 직접 학습 | 5순위 |

### 모델별 역할

```
PageRank / GNN
→ 개발자-프로젝트 관계망에서 영향력 계산
→ "이 개발자가 생태계에서 얼마나 중요한 노드인가"

BERT 파인튜닝
→ 커밋 메시지, PR 내용, 코드리뷰 텍스트 분석
→ 전문 분야 분류 (Backend / Frontend / AI·ML / ...)

LLM (API)
→ 분석 결과를 자연어 근거로 변환
→ "Backend 91점인 이유: Spring Boot PR 34건 머지..."

XGBoost 앙상블
→ 위 모델들의 결과를 합산
→ 최종 점수 및 프로파일 산출
```

### 학습 데이터 구축 전략

- GitHub API로 오픈소스 프로젝트 기여 데이터 대량 수집
- 레이블은 프로젝트 언어 / 토픽 태그로 자동 생성
- 별도 라벨링 작업 최소화

### 플랜 B (시간 부족 시)

1~3순위만으로도 제출 가능한 수준 확보
- BERT 파인튜닝 + PageRank + LLM 근거 생성
- GNN / XGBoost 앙상블은 시간 되면 추가

## DID / 블록체인 (확정)

| 항목 | 선택 |
|------|------|
| DID 메서드 | did:key |
| 블록체인 플랫폼 | Polygon 테스트넷 |

### 처리 흐름

```
VC 발급 요청
      ↓
did:key로 DID 생성 (로컬 키 쌍 생성)
      ↓
분석 결과를 VC 형태로 서명
      ↓
VC 해시값만 Polygon 테스트넷에 저장 (가스비 없음)
      ↓
verifyCredential() 호출 시
Polygon에서 해시 꺼내서 위변조 확인
```

### 선택 이유

- did:key: 블록체인 연동 없이 로컬에서 빠르게 DID 생성 가능
- Polygon 테스트넷: 가스비 없음, 이더리움 호환, 레퍼런스 풍부
- VC 전체가 아닌 **해시값만 저장** → 용량 최소화, 속도 빠름
- 추후 did:ethr로 마이그레이션 가능한 구조 유지

## 가중치 설계 (확정 — 추후 변경 가능)

### 최종 점수 산출 방식

활동 비율 기반 동적 가중치 적용 (옵션 C)

```
최종 점수 = 기여자 점수 × 기여자 활동 비율
           + Maintainer 점수 × Maintainer 활동 비율
```

활동 비율은 GitHub 전체 활동 중 타인 프로젝트 기여(커밋/PR/리뷰) vs 본인 프로젝트 활동(커밋/PR/리뷰) 수로 자동 계산

예시:
```
타인 기여 70% / 본인 프로젝트 30%
→ 최종 점수 = 기여자 점수 × 0.7 + Maintainer 점수 × 0.3
```

### 기여자 점수 세부 가중치

| 항목 | 가중치 | 이유 |
|------|--------|------|
| PR 머지율 | 35% | 실질적 기여의 가장 직접적 지표 |
| 코드리뷰 품질 | 25% | 전문성 판단에 핵심 |
| Maintainer 승인 여부 | 15% | 프로젝트 핵심 인정 여부 |
| 기여한 프로젝트 규모 | 10% | 영향력 범위 |
| 기여 지속성 | 10% | 단발성 vs 장기 기여 |
| 이슈 해결률 | 5% | 보조 지표 |

### Maintainer 점수 세부 가중치

| 항목 | 가중치 | 이유 |
|------|--------|------|
| 프로젝트 채택률 | 30% | 프로젝트 영향력의 가장 직접적 지표 |
| 커뮤니티 활성도 | 25% | 외부 기여자 유입 = 프로젝트 신뢰도 |
| PR 리뷰 품질 | 20% | Maintainer 전문성 핵심 |
| 이슈 응답 속도 | 10% | 운영 신뢰성 |
| 릴리즈 지속성 | 10% | 유지보수 여부 |
| 문서화 수준 | 5% | 보조 지표 |
